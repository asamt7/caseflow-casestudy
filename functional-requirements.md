# Glossary — CaseFlow

| Term | Definition |
|---|---|
| **Artefact** | A piece of evidence, document, or record associated with a support case (e.g., a damage photo, a Jira ticket, a call transcript) |
| **AHT** | Average Handling Time — the average time an agent spends on a case from first contact to resolution |
| **Case** | A unit of work representing one customer support incident, spanning all related communications across channels |
| **Case Event** | A single communication or action on a case — normalised from a source channel and stored in the CaseFlow timeline |
| **Case Intelligence** | The AI-powered ability to query, summarise, and analyse a case using natural language |
| **Case Linking** | The act of associating a communication from an external channel to an existing CaseFlow case |
| **Channel** | A communication medium used in the support process — e.g., live chat, email, WhatsApp, Slack, Jira |
| **Context Switching** | The act of moving between different applications or tools to gather information — a key source of agent inefficiency |
| **Embedding** | A numerical vector representation of text, used to find semantically similar content in the vector store |
| **FCR** | First Contact Resolution — the percentage of cases resolved on the first interaction without escalation or follow-up |
| **Gap Detection** | The process of identifying missing artefacts or required evidence in a case based on case type rules |
| **Hallucination** | When an AI model produces information that sounds plausible but is not grounded in the actual input data |
| **Handoff** | The transfer of a case from one agent to another, requiring full context to be communicated |
| **LLM** | Large Language Model — the AI model used for natural language understanding and generation (Claude Sonnet) |
| **MoSCoW** | A prioritisation framework: Must Have, Should Have, Could Have, Won't Have |
| **NFR** | Non-Functional Requirement — a requirement describing how the system performs, not what it does |
| **Orchestration Layer** | A software layer that coordinates multiple underlying systems — CaseFlow's core architectural role |
| **PII** | Personally Identifiable Information — customer names, emails, order details that must be protected |
| **PoC** | Proof of Concept — a limited prototype built to validate the core value proposition before full development |
| **RAG** | Retrieval-Augmented Generation — an AI pattern where the model retrieves relevant context before generating a response, grounding its answers in real data rather than memory |
| **RBAC** | Role-Based Access Control — access permissions assigned by user role (Agent, Team Lead, Admin) |
| **RMA** | Return Merchandise Authorisation — a document authorising a customer to return a product |
| **Sentiment Analysis** | AI-powered detection of emotional tone in customer messages (Positive / Neutral / Frustrated / Escalating) |
| **SKU** | Stock Keeping Unit — a unique identifier for a product variant |
| **SLA** | Service Level Agreement — a commitment to respond or resolve within a defined timeframe |
| **SMX** | Source-to-Target Mapping — used in the Data Warehouse project context; not used in CaseFlow |
| **Source Citation** | In CaseFlow AI responses, the reference to the specific channel event that supports a given answer |
| **Timeline** | The chronological, unified view of all case events across all channels — CaseFlow's core UI component |
| **Vector Store** | A database optimised for storing and retrieving embeddings — used in CaseFlow's RAG pipeline |
| **Webhook** | An HTTP callback that sends real-time data from one system to another when an event occurs |
| **WISMO** | "Where Is My Order" — the most common type of e-commerce support enquiry |
